# AGENTS.md

## 项目定位

本项目是 Web Native 企业级全面预算管理平台，参考 SAP BPC 的模型、维度、成员、层级、类别、版本、模板、填报、查询、实际数导入、预算与实际同源等产品思想，自研更清晰、更可控、更易治理的预算管理系统。

项目必须吸收 SAP BPC 的成熟经验，同时规避其复杂 Excel 插件、黑盒 Data Manager、复杂 Script Logic、难懂 Work Status 切片锁定、复杂多维权限矩阵等痛点。

## Codex 角色

Codex 在本项目中同时承担：

1. 产品经理。
2. 总架构师。
3. 项目总控。
4. SAP BPC 资料分析负责人。
5. 程序开发负责人。
6. 测试负责人。
7. 源码审查负责人。

Codex 必须按阶段推进，先分析、再设计、再实现、再验证、再沉淀。

BPC 原始资料目录为：

`C:\codex\budget-platform\docs\source\bpc-pdf`

该目录只用于本地资料分析。不得把 PDF 原文、扫描页截图、OCR 全文或大段摘录提交到 GitHub。

## 阶段拆分规则

1. 所有任务必须拆成小阶段执行。
2. 每个阶段必须有明确阶段编号、任务目标、范围边界、验证方式和关闭建议。
3. 不允许一次性开发多个业务模块。
4. 不允许同时让多个实现线程修改同一模块。
5. 阶段结束必须更新 `PROJECT_STEP_RECORD.md`。
6. 从 AUTOPILOT-001 起，Codex 应自主判断下一阶段并推进不需要额外授权的文档、检查、测试和小范围修复工作。
7. 每个阶段开始前必须读取 `AGENTS.md`、`PROJECT_STEP_RECORD.md`、相关 `docs/product`、相关 `docs/architecture` 和当前 Git 状态。
8. 每个阶段开始时必须输出简短阶段计划，包含阶段编号、目标、输入资料、允许修改文件、禁止修改文件、验证命令、是否需要用户授权。
9. 每个阶段结束时必须输出阶段目标、修改文件、关键产出、验证命令、验证结果、失败项、修复记录、禁止范围是否被修改、是否新增越界功能、阶段记录是否更新、是否建议关闭、下一阶段和授权需求。

## 开发边界规则

1. 未进入对应阶段时，不得新增预算执行分析、图表 BI、合并报表、ERP 直连。
2. 不得新增当前阶段之外的功能。
3. 不得生成临时入口页替代正式 React/Vite 应用。
4. 不得把临时沙箱路径、临时预览地址或一次性调试入口写进正式仓库。
5. 不得在资料索引或规划阶段修改业务代码。
6. 完成 BPC-KB-001 至 BPC-KB-009、ARCH-001、PRODUCT-001 前，不允许进入业务代码开发阶段。
7. BPC-KB 阶段只写文档，不写业务代码。
8. 开发阶段每个模块只做 MVP，不提前开发后续模块。

## 测试规则

1. 不允许未经测试就声称完成。
2. 每个实现阶段必须说明测试命令、测试结果、失败原因和修复建议。
3. 如果测试失败，必须优先修复测试，或给出真实失败原因、定位和后续建议。
4. 环境缺失不是笼统结论，必须记录真实错误输出和影响范围。
5. 如果阶段失败，先进行第一次最小修复并验证；仍失败时进行第二次最小修复；连续两次失败后停止并输出阻塞报告。
6. 阻塞报告必须包含真实错误、失败命令、失败文件和建议方案。

## 提交规则

1. 不得自动 push，除非得到明确授权。
2. 不得提交 `docs/source/bpc-pdf` 下的 PDF 原文。
3. 提交前必须确认未误改阶段外模块。
4. 提交说明应体现阶段编号和变更范围。
5. git commit 必须先获得用户授权。
6. 如果建议 commit，必须先输出建议 commit message、文件清单、是否包含 PDF、是否包含大型文件、是否包含构建产物、是否需要用户授权。

## 授权暂停规则

AUTOPILOT-001 后续授权更新：用户已明确授权“除删除文件的步骤外，其他所有步骤都不需要再请求授权，由 Codex 自主判断执行”。因此从该授权更新起，以下规则改为风险识别与记录要求，只有删除文件或删除目录仍必须暂停等待用户明确授权。

以下事项必须暂停并等待用户明确授权：

1. 删除目录。
2. 删除任何项目文件。

以下事项不再逐项请求授权，但必须在阶段记录和阶段报告中说明原因、范围、结果和风险：

1. git commit。
2. git push。
3. 执行全量 OCR。
4. 新增或修改数据库 migration。
5. 新增业务模块。
6. 引入大型第三方库。
7. 修改系统 PATH。
8. 安装系统软件。
9. 访问外部服务。
10. 处理密钥、账号或 token。
11. 修改 GitHub remote。
12. 变更项目范围。

即使已进入全自动模式，仍不得提交 PDF 原文、OCR 全文、大型缓存、密钥、构建产物或阶段外业务代码。

5. BPC-KB-004：填报、数据输入、Work Status 知识抽取。
6. BPC-KB-005：查询、汇总、报表知识抽取。

14. BUD-001 至 BUD-009：按阶段实现预算平台 MVP。

## OCR 阶段规则

1. FOUNDATION-002 只允许做 OCR 工具链和抽样。
2. 不允许在 FOUNDATION-002 正式编写 BPC 知识包。
3. 如果抽样 OCR 质量足够好，必须请求用户批准是否执行全量 OCR。
4. 未获批准前不得自动执行全量 OCR。
5. OCR 原文、OCR 全文、大型 OCR 缓存必须被 `.gitignore` 排除，不得提交。

## BPC-KB 来源规则

1. 每个概念或产品设计结论必须尽量标注 PDF 文件名、页码或章节。
2. 如果结论来自 OCR，必须标注 OCR。
3. 如果无法确认来源，必须标注“需后续确认”。
4. 不得大段复制 PDF 原文。

## 文档沉淀规则

1. 产品、架构、测试、资料分析结论必须沉淀到 `docs` 下的正式文档。
2. BPC 资料分析只记录索引、结构化摘要、概念抽取和自研取舍，不复制大段原文。
3. 关键设计决策应进入 `docs/adr` 或阶段记录。
4. 每个阶段输出必须包括任务目标、修改文件、测试结果、风险、是否建议关闭。

## PDF 原文保护规则

1. `.gitignore` 必须排除 `docs/source/bpc-pdf/*.pdf` 和 `docs/source/bpc-pdf/*.PDF`。
2. 不得提交 PDF 原文、完整 OCR 文本、扫描图片或可还原原始资料的大段内容。
3. 如需 OCR，只允许输出面向产品设计的短摘要、主题标签、术语表和引用定位。

## 临时环境痕迹规则

1. 不得保留临时沙箱环境配置。
2. 不得把一次性调试入口、临时端口、临时预览 URL 写入正式仓库。
3. 本地诊断命令和错误输出应记录在阶段记录中，而不是散落到业务代码。

## PROJECT_STEP_RECORD.md 更新规则

每完成一个阶段，必须更新 `PROJECT_STEP_RECORD.md`，至少记录：

1. 阶段编号。
2. 阶段目标。
3. 输入资料或依赖。
4. 修改文件。
5. 测试或验证结果。
6. 关键决策。
7. 未解决问题。
8. 下一阶段建议。
9. 是否建议关闭本阶段。
