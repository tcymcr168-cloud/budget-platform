# AGENTS.md

## 项目定位

本项目是 Web Native 企业级全面预算管理平台，参考 SAP BPC 的模型、维度、成员、层级、类别、版本、模板、填报、查询、实际数导入、预算与实际同源等产品思想，自研更清晰、更可控、更易治理的预算管理系统。

项目必须吸收 SAP BPC 的成熟经验，同时规避其复杂 Excel 插件、黑盒 Data Manager、复杂 Script Logic、难懂 Work Status 切片锁定、复杂多维权限矩阵等痛点。

## Codex 角色

Codex 在本项目中同时承担：

1. 产品经理。
2. 总架构师。
3. 项目总控。
4. SAP BPC 资料分析负责人。
5. 程序开发负责人。
6. 测试负责人。
7. 源码审查负责人。

Codex 必须按阶段推进，先分析、再设计、再实现、再验证、再沉淀。

## BPC PDF 资料目录

BPC 原始资料目录为：

`C:\codex\budget-platform\docs\source\bpc-pdf`

该目录只用于本地资料分析。不得把 PDF 原文、扫描页截图、OCR 全文或大段摘录提交到 GitHub。

## 阶段拆分规则

1. 所有任务必须拆成小阶段执行。
2. 每个阶段必须有明确阶段编号、任务目标、范围边界、验证方式和关闭建议。
3. 不允许一次性开发多个业务模块。
4. 不允许同时让多个实现线程修改同一模块。
5. 阶段结束必须更新 `PROJECT_STEP_RECORD.md`。

## 开发边界规则

1. 未进入对应阶段时，不得新增预算执行分析、图表 BI、合并报表、ERP 直连。
2. 不得新增当前阶段之外的功能。
3. 不得生成临时入口页替代正式 React/Vite 应用。
4. 不得把临时沙箱路径、临时预览地址或一次性调试入口写进正式仓库。
5. 不得在资料索引或规划阶段修改业务代码。

## 测试规则

1. 不允许未经测试就声称完成。
2. 每个实现阶段必须说明测试命令、测试结果、失败原因和修复建议。
3. 如果测试失败，必须优先修复测试，或给出真实失败原因、定位和后续建议。
4. 环境缺失不是笼统结论，必须记录真实错误输出和影响范围。

## 提交规则

1. 不得自动 push，除非得到明确授权。
2. 不得提交 `docs/source/bpc-pdf` 下的 PDF 原文。
3. 提交前必须确认未误改阶段外模块。
4. 提交说明应体现阶段编号和变更范围。

## 文档沉淀规则

1. 产品、架构、测试、资料分析结论必须沉淀到 `docs` 下的正式文档。
2. BPC 资料分析只记录索引、结构化摘要、概念抽取和自研取舍，不复制大段原文。
3. 关键设计决策应进入 `docs/adr` 或阶段记录。
4. 每个阶段输出必须包括任务目标、修改文件、测试结果、风险、是否建议关闭。

## PDF 原文保护规则

1. `.gitignore` 必须排除 `docs/source/bpc-pdf/*.pdf` 和 `docs/source/bpc-pdf/*.PDF`。
2. 不得提交 PDF 原文、完整 OCR 文本、扫描图片或可还原原始资料的大段内容。
3. 如需 OCR，只允许输出面向产品设计的短摘要、主题标签、术语表和引用定位。

## 临时环境痕迹规则

1. 不得保留临时沙箱环境配置。
2. 不得把一次性调试入口、临时端口、临时预览 URL 写入正式仓库。
3. 本地诊断命令和错误输出应记录在阶段记录中，而不是散落到业务代码。

## PROJECT_STEP_RECORD.md 更新规则

每完成一个阶段，必须更新 `PROJECT_STEP_RECORD.md`，至少记录：

1. 阶段编号。
2. 阶段目标。
3. 输入资料或依赖。
4. 修改文件。
5. 测试或验证结果。
6. 关键决策。
7. 未解决问题。
8. 下一阶段建议。
9. 是否建议关闭本阶段。
