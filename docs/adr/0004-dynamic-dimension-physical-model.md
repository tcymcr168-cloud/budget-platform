# ADR-0004: Dynamic Dimension Physical Model

状态：Accepted

日期：2026-05-06

## 背景

预算模型需要支持系统维度和少量自定义维度。若为每个模型动态建表，会导致 migration、权限、查询、审计和模板解析复杂化；若全部用 EAV，则常见查询性能和可解释性会变差。

ARCH-001 已提出混合坐标模型：核心维度显式列，自定义维度使用轴表。

## 决策

采用混合坐标模型：

1. `fact_value` 保存核心维度显式列：Account、Entity、Time、Category、Version。
2. `fact_value_axis` 保存模型自定义维度坐标。
3. `model_dimension.axis_order` 定义完整坐标顺序。
4. `coordinate_hash` 使用模型 ID、核心维度成员、自定义维度成员按固定顺序生成。
5. 不为每个预算模型动态创建事实表。

## 理由

1. 核心维度显式列覆盖费用预算最常用查询路径。
2. 自定义维度轴表保留模型扩展能力。
3. 坐标 hash 支持导入幂等和重复校验。
4. 避免模板或模型变更带来动态 DDL。

## 后果

正向影响：

1. BUD-003 至 BUD-009 可以共享事实坐标规则。
2. 查询、填报、导入可以围绕同一事实层演进。
3. 物理模型既可解释，又保留自定义维度空间。

代价：

1. 自定义维度查询需要 join `fact_value_axis`。
2. 索引设计需在 BUD-002 和 BUD-008 继续细化。
3. 坐标 hash 的生成必须集中在服务层，不能散落到各模块。

## 非目标

1. 本 ADR 不创建 migration。
2. 本 ADR 不定义最终 DDL。
3. 本 ADR 不实现查询优化。
